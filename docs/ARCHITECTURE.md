# Architecture

Atlas is a Claude Code plugin. This document describes **what it's technically made of and how it works** (stack, components, flows, boundaries). File inventory is not here — it's in the mirror indices (`docs/commands/ABOUT.md` etc).

## Stack

| Layer | Tech |
|---|---|
| Manifest and metadata | JSON (`.claude-plugin/plugin.json`) |
| Documentation and instructions | Markdown with YAML frontmatter |
| Commands (slash) | Markdown prompts |
| Skill | Markdown with a frontmatter descriptor |
| Subagent | Markdown prompt with frontmatter (`tools:`, `description:`) |
| Hook | Bash script + JSON declaration (`hooks.json`) |
| Mirror resolution in the hook | Python 3 stdlib (`re`, `fnmatch`, `glob`) — optional; falls back to a generic reminder if python3 is missing |

No network calls, no database, no background processes. Runtime is the Claude Code process, which triggers either a bash script or a markdown read on events.

## Components (runtime roles)

1. **Skill** (`skills/atlas/SKILL.md`, ~60 lines) — lightweight context rule. Activates in atlas-style repos. Provides 5 fallback rules and **tells the agent to read the repo's AGENTS.md** — that's where per-repo methodology and mapping live.
2. **Command** — slash entry point. `/atlas:init` (heavy generator: scans code, writes AGENTS.md and docs/) and `/atlas:sync` (audit + one-confirmation fix).
3. **Subagent** — `atlas-auditor`, separate context, read-only. Resolves mirrors, returns an Undocumented/Orphan/Drift report. Invoked from `/atlas:sync`.
4. **Hook** — `remind-doc.sh` on PostToolUse. After every code edit, resolves the edited path via mirrors and prints "update matching doc: X" into the agent's context.

Key principle: the methodology lives in the **user's repo** AGENTS.md (generated by init), not in the skill. The skill supplies only minimum runtime behavior. The plugin is an orchestrator: form (init) + maintain (sync).

## Data flows

The only persistent state is the files in the repository. Four key flows:

### Flow 1: install

```
User:   /plugin install atlas
Claude: reads .claude-plugin/plugin.json
        loads commands/, skills/, agents/, hooks/
        registers PostToolUse hook
User:   /reload-plugins
```

### Flow 2: `/atlas:init` — bootstrap

```
User:   /atlas:init
Agent:  reads commands/init.md
Agent:  asks project name, language, purpose, audience
Agent:  scans code: package.json / pyproject.toml / go.mod / Dockerfile / ...
Agent:  infers stack, full code tree, proposes Code↔docs mapping
Agent:  shows the file list, waits for apply/drop/abort
Agent:  generates AGENTS.md (with real content from the scan), docs/ARCHITECTURE.md,
        docs/ABOUT.md, full mirror of the code tree
Output: "atlas: initialized — N files, next: read AGENTS.md"
```

### Flow 3: code edit → reminder

```
Agent:  Edit foo.py
Claude: fires PostToolUse hook → hooks/remind-doc.sh
Script: gate check (AGENTS.md + docs/ in cwd?), skip filter, then
        python3: read docs/**/*.md frontmatter, match foo.py against mirrors globs
Stdout: "[atlas] edited foo.py — update matching doc: docs/<path>/foo.md"
Agent:  reads the reminder, opens the matching doc, updates it, reports
```

### Flow 4: `/atlas:sync` — maintenance

```
User:   /atlas:sync [--check]
Agent:  reads commands/sync.md
Agent:  Task(atlas-auditor, prompt="audit this repo")
Auditor (fresh context):
        enumerate code files, parse mirrors, diff vs filesystem
        return structured report (Undocumented / Orphan / Drift)
Agent:  print report
        if --check: stop
        else: build edit plan, ask ONE confirmation (y / pick / n), apply
```

## Boundaries

- Atlas does **not** make network calls.
- Atlas does **not** run background processes (everything is synchronous, on-trigger).
- Atlas does **not** write files outside `docs/` (except what the user explicitly confirms in `/atlas:sync`).
- Atlas does **not** auto-generate docs from code — the source of truth for docs is human/agent hands; the plugin reminds and audits.
- Atlas does **not** enforce the discipline in non-atlas repos: every component gates on the presence of both `AGENTS.md` and `docs/`.
- Atlas does **not** work with non-markdown docs (PDF, Notion, Confluence are not supported).

## Invariants

1. **Methodology lives in the user's AGENTS.md, not the skill.** The plugin does two things: `init` writes methodology into the repo; `sync` audits against it. The skill (~60 lines) is just the minimum rules + pointer to AGENTS.md.
2. **`mirrors:`-frontmatter is the mapping source of truth for parsers** (auditor, hook). The Code↔docs table in AGENTS.md is for humans. On conflict, frontmatter wins.
3. **Repo root = plugin root.** `.claude-plugin/plugin.json` at the top level, no nested `plugins/<name>/`. Consumable from any marketplace via `{"source": "github", "repo": "lzvsk/atlas"}`.
4. **Plugin silent in non-atlas repos.** Skill doesn't activate, hook is silent, commands report "not an atlas repo".

## Dependencies

**Runtime, required:**
- Claude Code (any version with hooks.json and subagent support).
- Bash 3.2+ (macOS default works).

**Runtime, optional:**
- Python 3 (for mirror resolution in the hook) — without it the hook emits a generic reminder.
- `jq` (for parsing tool_input JSON in the hook) — without it the hook falls back to sed.

**Dev-only:** none. The plugin is markdown + bash; nothing to build or test through a build system.

## Deployment

None. Atlas is a client-side plugin. Installation goes through `/plugin install atlas` in Claude Code, which clones the repo locally for the user.
